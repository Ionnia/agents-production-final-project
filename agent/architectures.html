Смысл: чистый агентный конвейер — load_context префильтрует офферы, специалисты дают структурный анализ, supervisor собирает план, FinalJsonAgent гарантирует строгий enum, CriticAgent рефлексирует (заземляясь на validate как инструмент). Решения об исходе — агентные, без детерминированных гейтов.
Рубрика над B2: Action 2→3, Role 1→2, Reflection 0→1/2 + MAS-фрейминг (критерий 2/3). Связь с продом: MAS живёт внутри одного LangGraph-графа — не противоречит agent-service/openapi.yaml.
Проблемы и лечение (критерий 5): (1) наивный MAS терял контекст между агентами → брифуем специалистов контекстом группы; (2) переэскалация — policy_node подмешивал «визовые ограничения» в каждый RAG-запрос → эскалация привязана к конкретному риску участника; (3) бюджет из запроса («уложись в 160000») извлекает IntentAgent.
Вывод: улучшения подняли качество, но B3 даёт выше entity, ниже outcome, чем B2 — решения принимаются дальше от данных. Для этой задачи дисциплина B2 ≥ чистого MAS.
Файл: baselines/mas_supervisor_baseline.py

Смысл: B1–B3 показали — entity рождается у специалистов, outcome у заземления на надёжные факты. Final берёт обе сильные стороны: специалисты дают id (entity ≈ 1.0), а детерминированный калькулятор feasibility (инструмент, не решение) считает стоимость и бюджет точно; решают агенты (supervisor/critic) по этим фактам и порядку, по умолчанию clarification при «не влезает», rejection — только при явном запрете компромисса. Это паттерн B2 (факты — tool, решение — агент) + специалисты B3.
Почему так, а не LLM-оценщик: ранний полностью-агентный FeasibilityAgent переотказывал — LLM-вердикт о бюджете шумный. Калькулятор-инструмент вернул надёжный грунт; арифметику не отдаём LLM.
Связь с продом: это reasoning-ядро будущего agent-service/ (Contract A); обёртка stateful-threads + долгая память + HITL + реальный Contract B — следующий слой деплоя.
