# Проблемы и ограничения (критерий 5)

Кратко — что встретили при разработке агента (B1 → B2 → B3 → Final) и куда развивать.

## С чем столкнулись

1. **LLM-арифметика ненадёжна.** Агент-оценщик бюджета переотказывал (выдумывал перерасход). Лечение:
   стоимость/бюджет считает **детерминированный калькулятор-факт** (инструмент), а решение об исходе —
   агент. Арифметику LLM не отдаём.
2. **MAS-координация.** Наивный supervisor терял контекст между агентами (искал «Стамбул» вместо кода
   `IST`) → ложный `rejection`. Лечение: брифовать специалистов готовым контекстом группы.
3. **Оверфит vs обобщение.** Правила в промпте вида «ребёнок ≠ escalation» — это подгонка под qa.
   Вынесли поведение в **доменный регламент** `data/documents/05_agent_behavior.md` (RAG), промпты —
   общие.
4. **LLM-вариативность.** На пограничных кейсах outcome скачет ~0.65–0.85 при `temperature=0` (один и
   тот же код, разные прогоны). Метрику честно считать как диапазон/среднее.
5. **Граница `rejection ↔ clarification`.** Конфликтная группа G-0005: gold у Q-005 — `rejection`, у
   Q-018 — `clarification` при почти одинаковой формулировке. Это граница в самой разметке, не логика.
6. **Связанность entity и outcome.** Ранние версии роняли `entity` при escalation/clarification.
   Развязали: id специалистов попадают в ответ независимо от типа исхода.
7. **Пробел в данных.** README предписывал `05_agent_behavior.md`, но его не было — дописали.

## Ограничения текущего Final

- Stateless: нет памяти между ходами и HITL; `validate`/feasibility — локальные, не реальный Contract B.
- Выбор «пакетный тур vs раздельная сборка» неоптимален (Q-003).
- Eval на 20 кейсах — мало; нет усреднения по прогонам и метрик по категориям.

## Куда развивать

- Stateful-threads + долгая память предпочтений (Memory), HITL-уточнения только по критичному.
- Реальный Contract B вместо локального калькулятора; вынос в сервис `agent-service/`.
- Больше QA-кейсов, прогон с усреднением, метрики отдельно по категориям.
