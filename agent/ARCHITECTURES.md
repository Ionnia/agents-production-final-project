# Agent Architectures

**Диаграммы (наглядно) — в [`architectures.html`](./architectures.html)** (открыть в браузере: 4
векторные схемы B1 → B2 → B3 → Final, легенда, таблицы). Этот markdown — короткий текстовый индекс.

Все варианты используют один eval-харнесс (`data/qa/qa.jsonl` → `baselines/evaluate_predictions.py`),
поэтому метрики сопоставимы. Текущее состояние модуля — в [`SPECIFICATION.md`](./SPECIFICATION.md).

| Вариант | Суть | Статус | Файл |
|---|---|---|---|
| **B1** | один ReAct-агент Tool+RAG, выдаёт один JSON | реализован | `baselines/llm_tool_rag_baseline.py` |
| **B2** | LangGraph `draft(LLM) → normalize[det] → validate[det] → replan` | реализован | `baselines/langgraph_plan_validate_baseline.py` |
| **B3** | MAS: supervisor + специалисты (агенты-как-инструменты) + LLM-критик | реализован | `baselines/mas_supervisor_baseline.py` |
| **Final** | синтез: entity-машинерия B3 + outcome, заземлённый агентом выполнимости | реализован | `baselines/final_agent.py` |

**Идея B2:** разделение труда — LLM рассуждает и выбирает, детерминированный код считает и
сверяет (`validate` = локальный стаб backend-контракта `/internal/plans/validate`), а петля даёт
агенту переиграть решение по результатам валидации. Бюджетную арифметику не отдаём LLM.

**Идея B3 (чистый агентный MAS):** структурный конвейер
`load_context[det] → IntentAgent → специалисты (flight/hotel/tour/policy, structured-анализ) →
Supervisor → FinalJsonAgent (строгий enum) → CriticAgent (+`validate` как инструмент) → (replan|end)`.
Решения об исходе **агентны, без детерминированных гейтов**: эскалация — только если PolicyAgent
нашёл конкретный визовый/документный риск у реального участника; бюджет из запроса извлекает
IntentAgent (не regex). Контраст к B2: B2 = агент + детерминированные guardrails, B3 = чистое
рассуждение. По метрикам B3 **сильнее по entity, слабее по outcome** — задокументированный размен.

**Идея Final (синтез B2+B3):** B1/B2/B3 показали разделение — **entity рождается у специалистов** (B3),
**outcome — у заземления на надёжные факты** (B2). Final берёт обе сильные стороны: специалисты дают id
(entity ≈ 1.0), а **детерминированный калькулятор `feasibility` (инструмент, не решение)** считает
стоимость (без двойного счёта пакетного тура), бюджет (из запроса приоритетнее бюджета группы),
укладывается ли план и структурные нарушения. **Решение принимают агенты** (supervisor/critic) по этим
фактам и общему порядку (info → escalation → recommendation-если-в-бюджете → clarification → rejection),
**по умолчанию clarification** при «не влезает», rejection — только при явном запрете компромисса. Это
паттерн B2 (факты — tool, решение — агент) + специалисты B3. Граф:
`context → intent → специалисты → feasibility[калькулятор] → supervisor → finalizer → critic`.

> Ранний полностью-агентный вариант (LLM-FeasibilityAgent) **переотказывал**: LLM-вердикт о бюджете сам
> по себе шумный и смещал в rejection. Калькулятор-инструмент вернул надёжный грунт уровня B2.

> **Проблема MAS (для критерия 5).** Наивный supervisor сначала **регрессировал даже на лёгких
> кейсах**: терял контекст между агентами (вызывал специалиста без `group_id`), специалист искал по
> названию «Стамбул» вместо кода направления `IST` → пусто → supervisor ошибочно отдавал `rejection`.
> Чем больше агентов, тем больше точек рассинхрона. Лечение: брифовать специалистов готовым контекстом
> группы (код направления, даты, бюджет) и смягчить склонность supervisor к отказу. Вывод: MAS даёт
> роли/рефлексию, но требует дисциплины в передаче контекста, иначе проигрывает более простому B2.

## Метрики на `data/qa/qa.jsonl` (20 кейсов)

| Вариант | parse | outcome | entity |
|---|---|---|---|
| B1 | 1.00 | 0.85 | 0.667 |
| B2 | 1.00 | **0.95** | 0.867 |
| B3 | 1.00 | ~0.75 (шумит 0.65–0.85) | **1.0** |
| **Final** | 1.00 | **0.85** | **1.0** |

**Итог сравнения:** B2 — лучший outcome, но слабый entity; B3 — entity 1.0, но шумный outcome;
**Final — лучший баланс (0.85 + 1.0)**: специалисты дают entity, детерминированный калькулятор-факт +
агентное решение дают стабильный outcome. Для продукта важен именно баланс.

> Остаточные промахи Final (3): Q-005 и Q-018 — группа G-0005 с конфликтом требований, где сама
> разметка на грани (`rejection` vs `clarification` при почти одинаковой формулировке); Q-003 — выбор
> «тур vs раздельно». Это граница/разметка, а не логика — материал для критерия 5, не для подгонки.
