Продакшн-версия: final_agent_ranked внутри agent-service. От фронта «КудаЕдем» до LangGraph-графа на GigaChat. Принцип: факты считает код, решение принимает агент.

1Системный ландшафт (три сервиса)

Фронт говорит только с бэкендом. Бэкенд (BFF) владеет данными и драйвит агента по Contract A. Агент тянет бизнес-данные обратно по Contract B (Variant B) и предлагает черновик плана — сохраняет его бэкенд.

2Граф агента (сердце системы)

resolve [det]

Определяет тип запроса (подбор / вопрос о правилах / болталка) и для чата без группы вытаскивает направление, город вылета, даты и бюджет из всей переписки (каталог + LLM-извлечение фактов).

специалисты (LLM)

Flight/Hotel/Tour агенты ранжируют реальные офферы (ranked_candidates со score). Код берёт первый id, который существует в данных — защита от галлюцинаций.

feasibility [det]

Детерминированный калькулятор: суммарная стоимость (без двойного счёта пакетного тура), бюджет (из запроса приоритетнее), нарушения и атрибуты выбранного плана (прямой? звёзды? багаж?). Факты, не решение.

supervisor / critic (LLM)

Решают тип исхода по фактам и порядку (info → escalation → recommendation-если-в-бюджете → clarification → rejection). Критик ловит рассогласование и отправляет на переделку (до 2 раз).

3Жизненный цикл прогона и события (Contract A)

Бэкенд открывает run, агент стримит семантические SSE-события. Исход графа маппится в события, которые бэкенд валидирует и превращает в план/сообщение/вопрос для фронта.

Исход агентаSSE-события Contract A
recommendationplan_status: building → plan (DraftPlan: id + точки карты) → plan_status: ready → run_status: completed
clarificationclarifying_question (закрытые варианты + свободный ввод) → run_status
rejectionconstraints_conflict + suggested_relaxations → run_status (outcome=constraints_conflict)
escalationescalation (reason) → run_status
infomessage → run_status

Гарантия безопасности плана: перед recommendation сервис вызывает у бэкенда POST /internal/plans/validate (Contract B) и никогда не fail-open — если валидация недоступна или план невалиден, исход понижается до clarification. Так рекомендация всегда проходит финальную проверку бэкенда при сохранении.

4Ключевые принципы

Факты — код, решение — агент

Арифметику бюджета и проверки считает детерминированный калькулятор (LLM врёт в числах). Тип исхода выбирает LLM на основе этих фактов. Это дало стабильный outcome.

Entity ⟂ outcome

Найденные специалистами id переносятся в ответ независимо от типа исхода — поэтому entity = 1.0 даже при clarification/escalation.

Ranked-выбор офферов

Продовый вариант доверяет ранжирование офферов LLM (тоньше понимает «комфортный», «без лишних пересадок»); код лишь отсекает несуществующие id.

Доменные знания в RAG

Правила эскалации/уточнения/багажа/детей — в policy-документах (Chroma), а не зашиты в промпт. Это убрало подгонку под тест-кейсы.

Чат без группы

Направление/вылет/даты резолвятся из переписки по каталогу; если неоднозначно — закрытый вопрос с вариантами, а не зацикливание.

Память

Рабочая память — весь транскрипт в один проход графа. Долговременная — MemoryAgent пишет устойчивые предпочтения в группу через Contract B.
