Задача

Подобрать поездку под группу, бюджет и ограничения — и перестроить при изменении вводных

  • Выбрать перелёт + отель + тур с учётом состава группы, бюджета, предпочтений и правил сервиса.
  • Уметь уточнять недостающее, отказывать при невыполнимом, эскалировать визовые риски.
  • Две метрики: outcome_accuracy (правильный тип ответа) и entity_accuracy (правильные id).
2 / 12
Подход

Не один прототип, а прогрессия бейзлайнов — каждый шаг измеряю и делаю осознанное изменение

B1 → B2 → B3 → Final → Final-ranked. Один и тот же eval-набор (data/qa/qa.jsonl), поэтому шаги сравнимы. Каждое изменение — ответ на конкретную слабость предыдущего.

Бейзлайн B1

Один ReAct-агент: Tool + RAG

  • GigaChat с инструментами по CSV (перелёты/отели/туры) + RAG по правилам.
  • Сам решает, какие tools звать, выдаёт один JSON-ответ.
  • Слабость: нет самопроверки — бюджет и ограничения «на совести» модели.

Бейзлайн B2

LangGraph: агент предлагает → validate проверяет → replan

  • Добавил детерминированную проверку бюджета/ограничений (калькулятор-факт, не LLM — арифметику модели нельзя доверять).
  • Петля: нарушение → агент переигрывает решение.
  • Вывод: заземление на факты резко поднимает outcome, но entity всё ещё слабый (одиночный агент промахивается по id).

Бейзлайн B3

Мультиагент (MAS): supervisor + специалисты

  • Отдельные агенты по перелётам/отелям/турам/правилам → entity взлетел до 1.0.
  • Но outcome просел (~0.75): supervisor решает по текстовым сводкам, далеко от данных → шум.
  • Честный вывод: сложность не бесплатна. Специалисты дают entity, но решение нужно держать у фактов.

Метрика

outcome ~0.75 · entity 1.0

6 / 12
Final — синтез

Лучшее из B2 и B3: entity от специалистов + outcome на фактах

  • Специалисты находят id (entity ≈ 1.0), детерминированный feasibility считает факты (стоимость, бюджет, нарушения, атрибуты плана).
  • Supervisor/Critic выбирают исход по этим фактам и порядку решения; по умолчанию clarification, rejection — только при явном запрете компромисса.
  • Доменные правила (эскалация/дети/багаж) вынес в RAG-документ, а не в промпт — убрал подгонку под кейсы.

Метрика

outcome 0.90 · entity 1.0 — оба высоких

7 / 12
Final-ranked — продакшн

Финальный выбор: специалисты ранжируют офферы, а не просто фильтруют

  • Specialist отдаёт ranked_candidates (отсортированный список со score), код берёт первый реально существующий id (защита от галлюцинаций).
  • Тоньше понимает нюансы: «комфортный», «без лишних пересадок», «не премиум» — там, где keyword-фильтр Final промахивался.
  • Если ранжирование пустое/выдуманное — откат на детерминированный выбор. Надёжность сохранена.

Проблемы и ограничения · критерий 5

С чем столкнулся

  • LLM-арифметика ненадёжна → бюджет считает детерминированный калькулятор, не модель.
  • Оверфит vs обобщение → доменные правила ушли в RAG-документ, промпты остались общими.
  • LLM-вариативность на пограничных кейсах (outcome пляшет при temperature=0) → честно мерить как диапазон.
  • Граница reject ↔ clarification — местами спорна в самой разметке.
  • Развитие: persistent-память графа, больше QA-кейсов, метрики по категориям, убрать остатки CSV-fallback.

Итог

Зрелость по рубрике и вывод

  • Reasoning — недетерминированная цепочка + переплан по фактам.
  • Reflection — критик сверяет исход с фактами.
  • Action — агенты как инструменты, структурные вызовы.
  • Memory — транскрипт + долговременные предпочтения через Contract B.
  • Domain — RAG по правилам сервиса.

Главный принцип, который всё держит:
факты считает код — решение принимает агент.

Это дало одновременно высокий outcome и entity, без подгонки под тест.
